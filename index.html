Why Is PowerShell So Dangerous?

• Windows built-in, Microsoft-signed binary — cannot be removed
• Fileless — runs in memory without leaving any trace on disk
• AMSI, ScriptBlock Logging can be disabled (15+ methods)
• Over the network download + in-memory execute in one line
• COM, .NET, WMI, LDAP, Azure AD — access to entire Windows stack
• PS v2.0 logging mechanism is absent (downgrade attack)
• Living off the Land — legitimate appearance, low AV score
• Mandiant: all APT groups 94% use PS
Risk Surface in Enterprise Environment

• WinRM/PS Remoting with direct access to Domain Controllers
• Active Directory integration — BloodHound, SharpHound PS-based
• GPO with enterprise-wide single-command distribution
• SCCM/WMI with batch execution (10K+ endpoint)
• Azure AD, Exchange, SharePoint — cloud pivot
• Mimikatz, Empire, Rubeus — full toolkit available in PS
• Scheduled Task + Registry + WMI = 3 different persistence vectors
• IBM X-Force: Average dwell time 10 days in PS-based attacks

Event ID 4104 Fragmentation
Very long or obfuscated PS scripts don't fit in a single 4104 event. Windows splits the script into multiple event records and her kayda MessageNumber / MessageTotal adds fields. Attackers deliberately write very long scripts to evade analyst scrutiny.
🔍 Fragmentation Mechanics
Why Fragmentation Occurs? Windows Event Log applies a maximum ~32 KB data limit per single record. Long obfuscated PS scripts exceed this limit and system script MessageNumber = splits as 1/N, 2/N, 3/N ...

Attacker's Advantage: When SOC analysts evaluate each 4104 event independently, a fragmented event appears as just a meaningless base64 block and can be skipped.
# Event XML structure (fragmented 4104 example): <Event xmlns='...'> <System> <EventID>4104</EventID> ... </System> <EventData> <Data Name='MessageNumber'>2</Data> ← Bu 3'nd fragment <Data Name='MessageTotal'>3</Data> ← Tototal of 3 fragments <Data Name='ScriptBlockText'>[FRAGMENT 2 CONTENT]</Data> <Data Name='ScriptBlockId'>a1b2c3d4-...</Data> ← All fragments share same GUID <Data Name='Path'></Data> </EventData> </Event> # Critical alan: ScriptBlockId — SAME across all fragments of the same script # Bu GUID is the key to reconstructing the fragments

"Fileless Lie" — Add-Type + csc.exe (T1027.004)
PowerShell'in Add-Type cmdlet is used to access Windows APIs via C# code. Despite appearing fileless, csc.exe (C# compiler) is called in the background and temporary .cs and .dll files are written to %TEMP%. Incident responder can find these files in MFT. Ref: MITRE T1027.004

EDR Unhooking and D/Invoke with Syscall Bypass (T1562.001 + T1055)
Modern EDR'lar user-mode'da ntdll.dll hooks functions (with JMP instruction) to intercept malicious API calls. Advanced attackers these hooks D/Invoke (Dynamic Invoke), manual ntdll reload or indirect syscall bypasses with techniques. PS can execute C# code via Add-Type or reflection for these attacks. Ref: MITRE T1562.001 · TheWover D/Invoke Blog
🧬 Deep Technical Analysis
EDR User-Mode Hook Mechanism:
EDR agent, at system startup ntdll.dll's inner critical functions (NtAllocateVirtualMemory, NtWriteVirtualMemory, NtCreateThreadEx etc.) at the start of JMP [EDR_hook_handler] writes instruction. This way every API call first goes through EDR.

Bypass Method 1 — Manual ntdll Reload:
Attacker clean disk C:\Windows\System32\ntdll.dll reads file and copies to memory. API calls are made through this copy — EDR hooks stay disabled.

Bypass Method 2 — Direct/Indirect Syscall:
Direct kernel syscall numbers (SSN) without going through ntdll.dll. Applied from PS via Add-Type or reflection with C# → P/Invoke → inline assembly.

Bypass Method 3 — D/Invoke (Dynamic Invoke):
Developed by TheWover D/Invoke framework dynamically calls Windows APIs without writing to Import Address Table (IAT). Function pointer obtained by parsing PE header — IAT-based hooks are bypassed.

🎯 WMI Provider Architecture — Hijack Surface
Normal Query Flow:
Get-CimInstance Win32_Process → WMI Service (winmgmt) → Looks up: HKLM\...\CLSID\{...}\InprocServer32 → Loads cimwin32.dll (signed, legitimate) → Returns process list

“Fake 4104” — ScriptBlock AST Spoofing (Log Poisoning) — T1562.001
Log-deletion and thread suspension create detectable anomalies. Tier-1 actors do not stop logging — they poison it. Using the PowerShell AST API from C# or [ScriptBlock]::Create(), an attacker builds a hybrid ScriptBlock: a spoofed Extent feeds Get-Process into Event 4104, while a hidden EndBlock executes Invoke-Mimikatz. AMSI and Script Block Logging both read only the Extent — the executed code lives in EndBlock, invisible to both. SIEM receives valid timestamped 4104 events with benign content.

Architectural Denial of Service: ETW Session Starvation — T1562.006
ETW memory patching is risky and detected. Instead, attackers exploit a Windows architectural limit: the OS supports at most 64 simultaneous ETW trace sessions by default. An attacker fills all 64 slots with empty, anonymous sessions in seconds. When PowerShell or an EDR then tries to open a new session to record telemetry, the StartTrace call silently fails with ERROR_NO_SYSTEM_RESOURCES (1450). No memory violation, no hook removal, no alarm — just architectural starvation. The attack window is brief but sufficient for the malicious command to run unlogged.

Windows default: 64 maximum concurrent ETW trace sessions (user + kernel)

Victims: EDR telemetry sessions, PS Script Block Logging ETW session, any process trying to create a new trace session during the starvation window

Pure In-Memory Eventing: Timer Persistence — T1053 alternative
Standard persistence (Run Keys, Scheduled Tasks, WMI Event Subscriptons) all leave on-disk or registry traces that EDRs and Autoruns can enumerate. Using Register-ObjectEvent and System.Timers.Timer, an attacker creates a RAM-only recurring backdoor that fires every N seconds for the lifetime of the server uptime. No WMI subscription, no Task Scheduler entry, no registry key, no disk write. The only evidence is in the live PS runspace — which vanishes on reboot or process exit.
Ideal for: air-gapped servers with high uptime, pre-ransomware dwell periods, long-term espionage where reboots are rare.
Used by: Observed in post-exploitation frameworks (PowerShell Empire, Cobalt Strike PS agents) as a session-keepalive beacon mechanism
Ref: MITRE T1053 (Scheduled Task/Job — PS alternative) · MS Docs: Register-ObjectEvent

PowerShell Desired State Configuration (DSC) Abuse — T1546.003 + T1047
DSC is a built-in PS feature used to maintain enterprise systems in consistent configuration. Attackers create malicious MOF (Managed Object Format) files and Local Configuration Manager (LCM) can spread to all machines on the network with SYSTEM privileges. Critical blind spot: Execution powershell.exe not through, WmiPrvSE.exe occurs through — the vast majority of standard PS log rules cannot see this attack.
Ref: MITRE T1546.003 · MITRE T1047
🎯 DSC Architecture & Attack Surface
Normal DSC Flow:
1. Administrator creates MOF file (Invoke-DscResource or Start-DscConfiguration)
2. MOF file is sent to LCM (WMI Provider: MSFT_DSCLocalConfigurationManager)
3. LCM, WmiPrvSE.exe applies configuration as SYSTEM under
4. PS logs: None — because powershell.exe never starts

Attacker Scenario:
• Malicious MOF injection by impersonating legitimate DSC pull server
• Using LCM refresh cycle (default 15 min) for backdoor distribution
• "Compliance enforcement" feature: LCM continuously re-applies malicious config (persistence!)
• Pull mode: If malicious MOF placed on DC's SMB/HTTP share, entire domain affected

"Ghost Threads" — Asynchronous PS Runspace (T1620 + T1059.001)
[PowerShell]::Create().AddScript($payload).BeginInvoke() malicious code started with, even if main PS process is closed, the thread continues to run in the background. Invisible in process list — can only be detected via memory analysis (thread stack analysis).
Ref: MITRE T1620 · MITRE T1059.001
🎯 Technical Analysis
Normal PS Execution: powershell.exe starts → script runs → process closes → trace remains

Async Runspace: powershell.exe starts → background thread created with BeginInvoke() → main process closes → background thread continues to run inside .NET runtime → even if the attacker closes the window, the C2 beacon stays active

Neden Detection Zor?
• Get-Process powershell → No result (process closed)
• Task Manager → No PS process
• Sysmon Event 1 (process create) → Cannot monitor after termination
• Thread, another .NET host process (e.g. msbuild.exe, installutil.exe) may be living inside
